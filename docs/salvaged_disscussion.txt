## The Core Problem

Your normalization is **rule-complete but coverage-incomplete**. The *logic* for normalizing dates, places, agents, and publishers is solid and well-tested. But you don't have a systematic way to answer:

- How many raw values are actually mapping successfully vs. falling through to low-confidence defaults?

- How do I close those gaps without manually inspecting thousands of records?

You've hit the ceiling of what hand-curated rules and alias maps can do alone. You need a **feedback loop** between the data and the normalization.

## Three Approaches

### Option A: Audit-First — Normalization Coverage Report

Build a diagnostic pass over the indexed data that produces a **coverage report** per field:

- For each normalized field (date, place, publisher, agent): count records by confidence band (0.0, 0.5, 0.8, 0.95, 0.99)
- Export the low-confidence values grouped by frequency (the place that appears 200 times unmapped is higher priority than one appearing once)
- Flag the "unparsed" dates, unmapped places, ambiguous agents

This gives you a **prioritized work list**. You already have `scripts/normalization/extract_unmapped_places.py` doing a version of this for places — the idea is to generalize it across all fields.

**Pros:** Simple, no LLM cost, immediately actionable
**Cons:** Still requires human/LLM effort to actually resolve the gaps

### Option B: LLM-Assisted Batch Normalization with Human Review

You already have the pattern for this with `generate_place_alias_map.py` — LLM proposes mappings, human reviews. Extend this to:

1. Run the audit (Option A) to find unmapped/low-confidence values
2. Feed batches of raw values to an LLM with the field's controlled vocabulary as context
3. LLM proposes `raw → normalized` mappings with its own confidence
4. Output goes to a **review queue** (CSV or simple UI) where you accept/reject/edit
5. Accepted mappings get added to the alias maps, pipeline re-runs

This is basically the `place_alias_cache.jsonl` → `place_alias_map.json` pattern, generalized to all fields.

**Pros:** Scalable, leverages what you've already built, human stays in the loop
**Cons:** LLM costs, review bottleneck, need to handle disagreements

### Option C: Validation Against External Authority Files

For agents and subjects specifically, you have authority URIs from MARC `$0` subfields and the enrichment pipeline (NLI, Wikidata, VIAF). Use these as **ground truth anchors**:

1. For records that have authority URIs — fetch the canonical form from the authority source
2. Compare your normalized value against the authority's canonical form
3. Where they agree: boost confidence. Where they diverge: flag for review
4. Use the verified mappings to train alias maps for records *without* authority URIs

This is particularly powerful for agents (where VIAF/NLI have canonical name forms) and subjects (where LCSH has controlled vocabulary).

**Pros:** Grounded in authoritative sources, builds trust, leverages existing enrichment pipeline
**Cons:** Not all records have authority URIs, external API rate limits, only covers agents/subjects well

**Start with A, then layer B and C on top.** The audit report is cheap, fast, and tells you exactly where to focus. Then use B (LLM-assisted mapping) for places/publishers where no authority exists, and C (authority validation) for agents/subjects where it does.

The key insight is that all three feed into the same output: **better alias maps and confidence scores** — which your pipeline already knows how to consume.

- The agent can batch related proposals ("approve all 8 Latin-to-English place mappings?")
